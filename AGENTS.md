# AGENTS.md

## Working Conventions
- 開発ツールの導入は `mise` を前提にし、まず `mise install` を実行する。
- Python を使うコマンドは `python` 直実行ではなく `uv run` 経由で実行する。
- 外部フィードや外部ページを確認したら、「設定値」と「実際に取得できた内容」を切り分けて報告する。
- 外部サービスの表示差分、RSS の実表示、ログイン有無の切り分けには Playwright を積極的に使う。調査が速く、実態確認に向いている。
- Playwright で確認した生成物や調査用ファイルは、明示的に必要な場合を除いてコミットしない。
- ローカルの絶対パス、認証情報、秘密鍵、個人用トークンは git に含めない。コミット前に `npm run check:safety` を通す。
- 秘密情報の検査は自作正規表現だけに頼らず、OSS の定番ツールを優先する。このリポジトリでは `gitleaks` を使い、`mise` で導入する。
- 安全性チェックの実体は `scripts/check-safe-commit.sh` とし、`mise run check-safety` または pre-commit hook から実行する。
- `.gitignore` は作業生成物を確実に除外できる状態を保ち、新しい調査用・ビルド用ディレクトリを作ったら必要に応じて更新する。

## Pull Requests
- PR タイトルと本文は日本語で書く。
- Issue タイトルと本文も日本語で書く。
- PR 本文には最低限、`概要`、`変更内容`、`テスト`、`確認したこと`、`関連Issue` を含める。
- `テスト` には「何を実施したか」と「結果がどうだったか」を具体的に書く。
- PR 作成前には、Issue 起票、実装、テスト実行、テスト結果の PR 本文反映までを一連のフローとして完了させる。
- PR 作成前には `mise run prepr` を実行し、少なくとも安全性チェックと関連テストが通っていることを確認する。
- レビュー対応時は、対応有無の判断理由を明確にし、対応した場合は追加したテストも書く。
- 公開リポジトリでは GitHub の secret scanning / push protection の活用も前提にし、ローカルチェックと二重化する。

## Implementation Notes
- RSS だけでなく Atom フィードも考慮して実装・テストする。
- フィードパーサのようなフォーマット依存の処理は、サンプル文字列で再現できるテストを追加する。
