# PR チェックリスト

## 作成前
1. 対象 Issue を確認する。既存 Issue 対応では `gh issue view <番号>` を先に実行する。
2. 実装と関連テストを完了する。
3. `mise run prepr` を実行し、安全性チェックと関連テストが通ることを確認する。
4. Issue を閉じるのは、対応 PR のマージ後に `Closes #<番号>` で自動クローズさせるか、明示的に指示された場合だけにする。

## 本文ルール
- PR タイトルと本文は日本語で書く。
- 本文には `概要`、`変更内容`、`テスト`、`確認したこと`、`関連Issue` を含める。
- `テスト` には実行した内容と結果を具体的に書く。
- テスト結果は、実際に実行したコマンドと成功条件が分かる形で反映する。

## レビュー対応
- 対応有無の判断理由を明確に書く。
- 対応した場合は、追加したテストや再確認した内容も書く。
- レビュー前に Issue を完了扱いしない。実装確認が済んでいても、PR 未作成・未レビュー・未マージの状態では Issue は open のまま維持する。

## 公開リポジトリ
- GitHub の secret scanning / push protection を前提にする。
- ローカルの安全性チェックと GitHub 側の検査を二重化する。
